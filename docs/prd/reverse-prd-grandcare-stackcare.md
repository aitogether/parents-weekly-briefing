## 一、核心对比表

| 维度 | GrandCare Systems | StackCare（经 Connected Caregiver 销售） |
|------|-------------------|----------------------------------------|
| 核心承诺对象 | 养老机构 / 家属 | 子女 / 家属 |
| 数据源 | 传感器 + 健康测量 + 用药提醒 + 视频通话 | 室内运动传感器 + 温度（纯 IoT） |
| 输出频率 | 实时告警 + Dashboard 周视图 | 实时告警 + 每日 Daily Update |
| 子女端体验 | Web Portal 专业 Dashboard，图表很多 | App 内 Daily Update 通知 + 简单 Dashboard |
| 聚合层级 | 多维数据可视化，但仍偏 raw_data / 图表 | daily_digest，给一句「一切正常 / 有异常」 |
| 行动建议 | 无明确「建议你做 XX」 | 无，只有异常提示 |
| 客户与定价 | B2B 为主，$150–300/月（含硬件） | $75/月 + $150 注册费，C 端可买但价格高 |
| 生态位 | 综合平台：机构级「监控中台」 | 轻量被动监测：居家 IoT 安全层 |

## 二、GrandCare 反向 PRD 要点（我们不这么做）

### 不把子女当成「小号护理站」
GrandCare 的 Care Portal 面向的是专业护理团队，家属只是顺带访问者。父母周报 P0 必须明确：**第一使用者就是子女**，信息密度和语言风格都按非专业人士设计。

### 不把「看懂 10 张图」当成果
GrandCare 堆积大量图表和趋势线，却没有一句「本周总体状况如何」。父母周报要反向：**先有一段自然语言结论，再允许少量图表 drill-down**。

### 不把「多数据源」仅停留在展示层
GrandCare 确实集成了运动 + 健康设备 + 用药，但每一块都是各看各的。我们在 P0 就要把多源数据变成**同一结论句子的一部分**（例如：「活动下降 + 血压波动 → 建议这周问问睡眠 / 心情」）。

### 不把产品只做给 B 端机构
GrandCare 的定价和部署模式天然偏向养老院 / 机构，C 端家庭只是「顺带受益」。父母周报 P0：**只做 C 端子女订阅，不碰机构采购逻辑**。

## 三、StackCare 反向 PRD 要点（我们不这么做）

### 不把「只有 IoT 传感器」当全集
StackCare 的 Daily Update 很接近我们想要的形式，但数据源只覆盖「活动量 / 出入」。父母周报 P0 不能再造一个「活动日报」，至少要从一开始就设计成**可以插更多数据源的壳**（用药 / 通讯 / 消费）。

### 不输出「一切正常」四个字就打发
StackCare Daily Update 核心就是一句「今天一切正常」或类似提示。我们要反向：**即便是「绿灯周」，也要给子女一条具体可做的动作**（比如「这周可以放心，只是活动略降，建议多问问最近天气 / 关节疼不疼」）。

### 不把价格结构锁死在「硬件+高月费」
$75/月 + 注册费，对中国子女来说几乎没有可能普及。父母周报 P0 要从**纯 SaaS / 轻数据**入手，避免一上来就跟硬件绑定。

### 不忽视情绪层的文案
StackCare 的表达是典型工程语气：状态正常 / 活动减少。父母周报从 Day 1 就要写给「guilt eating me alive」那类子女——**文案要同时照顾信息 + 情绪安抚**。

## 四、风险点评（写给未来自己的警告）

1. 如果我们继续往「加更多图表、更多指标」的方向卷，就会变成一个小号 GrandCare——**专业，但子女不想每天点开**。
2. 如果我们只做「活动监控 + 一句一切正常」，那就是 StackCare 翻版，**没法承接用药 / 防骗 / 通讯这些更重的场景**。
3. B 端 / 机构化的诱惑很大（钱好收、票据好看），但一旦架构按机构需求设计，C 端子女的体验就会永远排在第二优先级。
4. 「多数据源整合」如果只停在 UI 层（多个 tab），技术上很容易交差，但业务上等于没整合——**真正的整合是同一条周报里的一个判断**。
5. 最危险的失败形态不是没人用，而是子女用了一段时间发现：**周报只是在提醒我焦虑，却没有帮我决定该做什么**——那我们会被归入「又一个数据 App」。
